The 2018 0CTF/TCTF competition was highly regarded for its advanced, realistic client-side challenges. The core theme across these challenges was bypassing modern Content Security Policies (CSP).The primary solutions for the three core XSS challenges from 0CTF 2018 are broken down below:1. bl0gCore Vulnerability: DOM XSS combined with Global Variable Overwrite / DOM Clobbering.The Defense: The target application implemented a strict modern CSP relying on 'strict-dynamic'. This meant that standard inline script injections or simple remote scripts were automatically blocked unless executed by an already-trusted script.The Bypass Strategy:The application contained an existing trusted script that dynamically created <script> elements by reading parameters from a configuration object (e.g., window.config.url).Attackers identified a variable overwrite vulnerability (or DOM clobbering via HTML tags with specific id attributes) that allowed them to overwrite the configuration property.By forcing window.config.url to point to a malicious JavaScript file, the site's own trusted script dynamically injected the attacker's script payload, effectively satisfying the 'strict-dynamic' policy requirement.2. h4xors.club2Core Vulnerability: Vulnerable postMessage handlers and Script Gadgets.The Defense: The challenge featured a strict nonce-based CSP containing script-src 'nonce-[value]' 'strict-dynamic' to prevent unauthorized inline script parsing.The Bypass Strategy:Cross-Origin Communication: The client-side code handled incoming postMessage requests without validating the origin (event.origin).Script Gadgets: This flaw allowed an attacker to embed the target site inside an iframe and pass custom JSON payloads to its event listener. The event listener processed the properties of the message payload and dynamically updated DOM elements (such as data.title).Execution: By feeding carefully formatted HTML text through the open postMessage interface, the payload triggered an execution path inside the application's existing framework files〞acting as a client-side "script gadget"〞to bypass the strict CSP restrictions entirely.3. h4x0rs.spaceCore Vulnerability: Appcache and Service Worker hijacking tied to a JSONP endpoint.The Defense: This site leveraged a strict layout where user inputs were aggressively HTML-escaped, blocking classic script injection sequences inside regular blog posts.The Bypass Strategy:SVG Upload: The application allowed users to upload SVG files, which were hosted same-origin but isolated from the principal page. While JavaScript could execute within the SVG context, it remained constrained by the CSP applied to the endpoint.JSONP Execution: Attackers utilized an available JSONP padding endpoint (/blog/pad.php?callback=...) to inject structured code blocks disguised as multi-line JavaScript comments (/* ... */).Persistence Hijacking: By combining the JSONP payload with the SVG context, they successfully registered a malicious Service Worker. Once registered, the worker intercepted all subsequent network traffic under that scope, enabling full persistence and data extraction despite the active CSP defenses.Summary of Defense TakeawaysThese challenges demonstrated that modern browser mechanisms (like CSP) cannot completely stop client-side threats if the application logic contains flaws. Secure implementations require:Strict origin verification for all window.postMessage listeners.Preventing DOM Clobbering by avoiding reliance on mutable global objects.Ensuring user-uploaded assets (like SVGs) are served from a distinct, sandboxed sandbox domain to restrict session token access.
